## Context

Фаза 1 закрыла генерацию и валидацию synthetic dataset v1, включая метаданные камеры и truth-данные по видимым звездам. Для фазы 2 нужно добавить минимальный распознаватель, который стабильно работает на clean synthetic и выдает воспроизводимый результат в виде pose + overlay + benchmark-метрики.

Технические ограничения:
- На данном этапе важнее воспроизводимость и диагностируемость, чем максимальная скорость.
- Контракты данных между симулятором и распознавателем должны оставаться явными и версионируемыми.
- Фаза 2 не закрывает domain gap; реалистичные оптические эффекты и real-world robustness вынесены в фазу 3.

## Goals / Non-Goals

**Goals:**
- Реализовать детерминированный baseline solver-пайплайн: detection -> pattern matching -> pose -> overlay/debug.
- Зафиксировать единый артефакт результата распознавания для одиночного кадра (гипотеза, confidence, pose, диагностические слои).
- Добавить benchmark harness для пакетного сравнения и автосохранения худших кейсов.
- Подготовить модульную структуру, позволяющую заменять алгоритмы внутри этапов без смены внешнего контракта.

**Non-Goals:**
- Работа на реальных фото, включая облачность, сильный noise и сложные оптические искажения.
- Мобильная оптимизация и работа в realtime.
- Продвинутые UI-функции; достаточно технического overlay/debug уровня.

## Decisions

1. **Явное разделение solver на 5 capability-модулей.**  
   Выбрано разделение на `solver-star-detection`, `solver-pattern-matching`, `solver-pose-estimation`, `solver-overlay-debug`, `solver-benchmark-harness`.  
   **Почему:** это соответствует roadmap и снижает риск смешивания эвристик распознавания с визуализацией/бенчмарком.  
   **Альтернатива:** монолитный модуль solver для ускорения старта.  
   **Почему не выбрано:** сложнее тестировать и локализовать деградации.

2. **Контракт промежуточных данных через typed pipeline payloads.**  
   Каждый этап принимает формализованный вход и возвращает формализованный выход (детекции, кандидаты матчинга, inliers/pose, overlay слои).  
   **Почему:** позволяет сохранить воспроизводимость и точечно валидировать этапы.  
   **Альтернатива:** передавать неструктурированные диагностические словари.  
   **Почему не выбрано:** повышает риск несовместимостей между этапами.

3. **Baseline matching c confidence + неоднозначность как первый класс результата.**  
   На уровне API результата фиксируется не только лучшая гипотеза, но и признак неоднозначности/низкой уверенности.  
   **Почему:** снижает количество ложных "уверенных" ответов и упрощает обработку отказов в следующих фазах.  
   **Альтернатива:** всегда возвращать top-1 без explicit confidence gating.  
   **Почему не выбрано:** ухудшает управляемость ошибок и интерпретируемость метрик.

4. **Pose estimation с робастной фильтрацией выбросов.**  
   Принят подход с robust fit (RANSAC-подобный или эквивалентный), чтобы устойчиво работать при частичных mismatch.  
   **Почему:** это минимальное требование roadmap к стабильной оценке ориентации.  
   **Альтернатива:** прямой least-squares без фильтрации.  
   **Почему не выбрано:** высокая чувствительность к одиночным ошибочным соответствиям.

5. **Benchmark-first интеграция для фазы 2.**  
   Любое изменение solver должно быть прогоняемо через единый harness, который сохраняет агрегаты и худшие кейсы.  
   **Почему:** позволяет объективно принимать go/no-go по критериям фазы.  
   **Альтернатива:** ручные ad-hoc прогоны.  
   **Почему не выбрано:** нет воспроизводимой базы для регрессий.

## Risks / Trade-offs

- **[Риск] Слабая переносимость с clean synthetic на реалистичные условия** -> **Mitigation:** явная фиксация границы фазы 2 и подготовка benchmark baseline для фазы 3.
- **[Риск] Ложные срабатывания при слабых детекциях** -> **Mitigation:** confidence gating и явная маркировка ambiguous-case в результате.
- **[Риск] Перекос метрик из-за ограниченного synthetic распределения** -> **Mitigation:** автоэкспорт worst-cases и периодический аудит с ручным просмотром.
- **[Trade-off] Модульность увеличивает объем интерфейсного кода** -> **Mitigation:** минимальные, но строгие контракты payload на этапе v1.
- **[Trade-off] Robust pose дороже вычислительно, чем простой fit** -> **Mitigation:** для фазы 2 приоритет качества над скоростью; оптимизация откладывается.

## Migration Plan

1. Добавить новые capability-спеки и согласовать требования acceptance-сценариями.
2. Реализовать этапы solver последовательно: detection -> matching -> pose -> overlay.
3. Подключить benchmark harness и зафиксировать baseline метрики на dataset v1.
4. Включить регулярный прогон benchmark как обязательную проверку перед переходом к фазе 3.

Rollback:
- Если интегрированный solver нестабилен, сохраняется возможность запуска отдельных этапов в debug-режиме без полного pipeline.
- Возврат к последнему стабильному baseline выполняется через pinned конфиг benchmark и версию solver-модулей.

## Fixed Parameters for Phase 2

1. **Worst-case serialization format (Option B): единый versioned формат уже в фазе 2.**  
   Принято использовать единый формат worst-cases c явной версией схемы, чтобы избежать миграции формата при переходе к фазе 3 и упростить построение tooling для анализа деградаций.

2. **Confidence acceptance policy (Option B): абсолютный порог + margin к второму кандидату.**  
   Принятие гипотезы выполняется только при одновременном выполнении условий:
   - `top1_conf >= T_abs` (начальное значение для MVP: `0.65`)
   - `top1_conf - top2_conf >= T_margin` (начальное значение для MVP: `0.10`)
   Если условия не выполнены, результат маркируется как ambiguous/no-accept.

3. **Go/No-Go thresholds for phase transition (Option B): сбалансированный gate.**  
   Для перехода к фазе 3 фиксируются целевые пороги на clean synthetic:
   - detection precision `>= 0.97`
   - detection recall `>= 0.95`
   - pose error: median `<= 0.4°`, P95 `<= 1.0°`
   - accepted-solve rate `>= 95%` (эквивалентно failure rate `<= 5%`)
   - reproducibility: повторный прогон с тем же seed/config дает расхождение агрегатов `<= 1%`

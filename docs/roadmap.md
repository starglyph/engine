# Дорожная карта

Этапы без привязки ко времени — в порядке технической зависимости.

## Фаза 0 — scope, данные и контракты

**Цель:** зафиксировать, что именно решаем, на каких данных и как измеряем успех.

- **0.1 Сценарии использования**
  - Определить обязательный режим: распознавание по одиночной фотографии.
  - Определить расширенный режим: учёт даты/времени и геолокации (из EXIF или вручную).
- **0.2 Критерии качества**
  - Зафиксировать метрики: угловая ошибка позы, полнота/точность детекции звёзд, время обработки кадра.
  - Установить целевые пороги для MVP и для последующих фаз.
- **0.3 Источники данных и лицензии**
  - Выбрать звёздный каталог и его подмножество для прототипа.
  - Выбрать источник линий созвездий и соглашение по наименованиям.
  - Проверить юридическую допустимость использования данных в проекте.
- **0.4 Форматы артефактов**
  - Описать формат synthetic-пакета: изображение, параметры камеры, truth-данные.
  - Разделить артефакты симулятора и распознавателя (каталоги, версии, форматы).
- **0.5 Базовая документация**
  - Добавить/обновить документы с форматом данных и методикой оценки.

**Критерий выхода из фазы:** формально зафиксированы метрики, форматы данных и источники с понятной лицензией.

## Фаза 1 — минимальный симулятор (dataset v1)

**Цель:** получать воспроизводимые синтетические кадры с эталонной разметкой.

- **1.1 Геометрия и камера**
  - Реализовать pinhole-проекцию и ориентацию камеры.
  - Экспортировать intrinsics/extrinsics для каждого кадра.
- **1.2 Рендер звёзд**
  - Преобразование величин звёзд в яркость пикселей.
  - Базовая модель PSF и ограничение динамического диапазона.
- **1.3 Базовые деградации**
  - Добавить шум сенсора (shot/read), лёгкий blur, JPEG-артефакты.
- **1.4 Генератор датасета**
  - Скрипт генерации train/val/test с фиксированным seed.
  - Сохранение метаданных по кадру и списка видимых звёзд.
- **1.5 Валидация симулятора**
  - Unit-тесты на корректность проекции.
  - Набор визуальных golden-примеров.

**Критерий выхода из фазы:** датасет `v1` собирается одной командой и воспроизводится при фиксированном seed.

## Фаза 2 — минимальный распознаватель (solver v1)

**Цель:** стабильно распознавать чистую синтетику и отдавать корректный оверлей.

- **2.1 Детекция звёзд**
  - Реализовать baseline-пайплайн детекции.
  - Измерять precision/recall на synthetic truth.
- **2.2 Matching звёздных паттернов**
  - Реализовать базовый алгоритм сопоставления (треугольники/квады или аналог).
  - Ввести оценку уверенности и фильтрацию неоднозначных кандидатов.
- **2.3 Оценка pose**
  - Оценка ориентации камеры по сопоставленным точкам.
  - Робастная фильтрация выбросов (например, RANSAC-подход).
- **2.4 Оверлей и отладка**
  - Рендер линий созвездий поверх кадра.
  - Добавить debug-слои: детекции, inliers/outliers.
- **2.5 Benchmark harness**
  - Скрипт прогона по датасету с отчётом по ключевым метрикам.
  - Автосохранение худших кейсов.

**Критерий выхода из фазы:** целевые метрики достигнуты на clean synthetic, а результаты повторяемы в benchmark-отчёте.

## Фаза 3 — приближение к реальности (domain gap)

**Цель:** сделать распознавание устойчивым к реальным оптическим и шумовым эффектам.

- **3.1 Оптические эффекты**
  - Добавить радиальную/тангенциальную дисторсию.
  - Добавить виньетирование и базовую хроматическую аберрацию.
- **3.2 Реалистичный шум**
  - Смоделировать засветку неба, hot pixels, motion blur.
- **3.3 Робастность алгоритма**
  - Ввести адаптивные пороги и устойчивость к частичной облачности.
- **3.4 Проверка на реальных фото**
  - Собрать небольшой набор фото добровольцев.
  - Подготовить процесс ручной/полуавтоматической валидации.
- **3.5 Регрессии и сравнение версий**
  - Сравнивать качество между версиями solver на synthetic + real eval set.

**Критерий выхода из фазы:** улучшения подтверждены на реальных фото без критической деградации скорости.

## Фаза 4 — продуктовый MVP (CLI/Web)

**Цель:** предоставить пользовательский поток "загрузка фото -> результат с оверлеем".

- **4.1 Интерфейс вызова**
  - Зафиксировать единый контракт входа/выхода анализатора.
- **4.2 Приложение-оболочка**
  - Реализовать CLI или простое веб-приложение.
  - Добавить отображение результата и диагностической информации.
- **4.3 Обработка отказов**
  - Явно объяснять причины неуспеха (смаз, пересвет, мало звёзд, облака).
- **4.4 Документация пользователя**
  - Quickstart, примеры, ограничения и рекомендации по съёмке.
- **4.5 Подготовка релиза**
  - Версионирование конфигов/моделей, changelog, минимальный релизный чеклист.

**Критерий выхода из фазы:** внешний пользователь может запустить анализ и получить интерпретируемый результат без чтения исходников.

## Фаза 5 — функциональные расширения

**Цель:** добавить ценность поверх базового сценария без потери стабильности.

- **5.1 Планеты и яркие объекты**
  - Подключить эфемериды и отрисовку планет при известной дате/времени.
- **5.2 Расширенные оверлеи**
  - Добавить подписи, режимы отображения и дополнительные справочные слои.
- **5.3 Улучшение UX-подсказок**
  - Добавить рекомендации пользователю по качеству кадра и условиям съёмки.

**Критерий выхода из фазы:** расширения работают как опциональные функции и не ухудшают базовые метрики распознавания.

## Фаза 6 — мобильный сценарий (R&D)

**Цель:** проверить реализуемость режима с низкой задержкой на мобильных устройствах.

- **6.1 Performance budget**
  - Зафиксировать целевые latency/FPS/энергопотребление.
- **6.2 Оптимизация инференса**
  - Исследовать ускорение на GPU/NPU, квантование и профили деградации качества.
- **6.3 Вариативность устройств**
  - Проверить работу на нескольких камерах/устройствах и подготовить fallback-режимы.
- **6.4 UX мобильного режима**
  - Калибровка, подсказки "куда навести камеру", оценка стабильности трекинга.

**Критерий выхода из фазы:** есть измеримый прототип на реальных устройствах с предсказуемым качеством и задержкой.

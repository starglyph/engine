# starglyph-serve — headless HTTP solve-сервис

Тонкий HTTP-фасад над `solve_frame_with_engine` (Этап 0 · C1): `POST` фотографии
неба → `SolveReport` JSON (+ по запросу — отрендеренный overlay-PNG). Ключевое
отличие от десктоп-приложения: базы паттернов **прогреваются на старте и
переиспользуются между запросами** — solve никогда не пересобирает то, что уже
лежит в диск-кэше.

Референс-реализация для self-host и фундамент РФ-беты (тонкий фронт E1, очередь
C2, контейнер C3 — поверх этого сервиса). Архитектурное решение «сервис живёт в
открытом `engine`» — ADR-013 в planning.

## Запуск

```bash
cd prototype
cargo run --release -p starglyph-serve            # 127.0.0.1:8080, дефолты ниже
```

Первый в истории запуск генерирует bootstrap-базу из HYG (~1 мин) и dense-полосы
слепой лестницы (22°/40°/65°) в `artifacts/cache`; они переживают рестарт —
дальше прогрев занимает секунды. `/readyz` отвечает `200`, как только bootstrap
прогрет на весь пул (dense-полосы догреваются в фоне, только в диск-кэш).

End-to-end проверка (поднимает сервис, решает реальный кадр, бьёт параллельными
запросами и проверяет, что базы не пересобираются):

```bash
make serve-smoke        # см. scripts/serve-smoke.sh
```

## API

### `POST /solve`

`multipart/form-data`:

| Поле | Тип | Описание |
| --- | --- | --- |
| `image` | файл | **Обязательное.** PNG/JPEG/BMP/TIFF; формат определяется по содержимому. EXIF (f35 → FOV-prior, дата → эпоха) подхватывается автоматически, как в CLI. |
| `fov_hint` | число | Горизонтальный FOV-prior в градусах, `[0.5, 120]`. Явный hint сильнее EXIF. |
| `epoch` | число | Эпоха наблюдения в дробных годах, `[1800, 2200]` (иначе выводится из имени файла/EXIF). |
| `utc_offset` | число | Смещение UTC наблюдателя в часах, `[-14, 14]`. |
| `no_exif` | bool | `true` — игнорировать EXIF-подсказки (чистый blind). |
| `grid` | bool | `true` — включить сетку RA/Dec в overlay-геометрию и рендер. |
| `overlay` | строка | `png` \| `inline` \| `none` (умолч.). Дублируется query-параметром `?overlay=...`; поле формы приоритетнее. |

Ответы (солв как операция всегда `200`, кроме перечисленного):

- **без `overlay`** → `200`, JSON `SolveReport` как есть (DTO из `contracts.rs`;
  `status: "solved" | "failed"`, `pose`, `fov`, `quality`, `timing_ms`,
  `detections`, `overlay`-геометрия в пикселях кадра).
- **`overlay=png`** → `200`, `image/png`: кадр с растянутой яркостью + линии
  созвездий, сетка, звёзды, планеты, маркеры детекций (рендерер общий с CLI,
  `starglyph_core::render`). Если кадр **не решился** — `422` с JSON-отчётом
  (рисовать нечего, отчёт с причиной отказа полезнее пустой картинки).
- **`overlay=inline`** → `200`, JSON `{ "report": SolveReport,
  "overlay_png_base64": "..." }` — отчёт и PNG за один solve (поле опущено,
  если кадр не решился).

Ошибки — единый конверт `{"error": {"code": "...", "message": "..."}}`:

| Код HTTP | `code` | Когда |
| --- | --- | --- |
| 400 | `missing_image`, `bad_image`, `bad_number`, `out_of_range`, `bad_bool`, `bad_overlay`, `bad_multipart` | битый запрос/картинка/параметры |
| 413 | `bad_multipart` | тело больше `--max-body-mib` |
| 503 | `not_ready` | bootstrap ещё прогревается (`/readyz` красный) |
| 503 | `busy` | ни один движок не освободился за `--queue-timeout-s` |
| 504 | `solve_timeout` | solve дольше `--solve-timeout-s` (движок вернётся в пул, когда брошенный solve доработает) |
| 500 | `internal` | паника solve-задачи и т.п.; движок заменяется свежим |

### `GET /healthz`, `GET /readyz`

`healthz` — живость процесса (всегда `200 ok`). `readyz` — `200 ready`, когда
bootstrap-база прогрета на весь пул; до того `503 warming`. `GET /` — краткая
справка по API.

## Конфигурация

Флаги CLI, каждый дублируется переменной окружения:

| Флаг | env | Умолчание | Описание |
| --- | --- | --- | --- |
| `--addr` | `STARGLYPH_SERVE_ADDR` | `127.0.0.1:8080` | адрес слушателя |
| `--catalog` | `STARGLYPH_SERVE_CATALOG` | первый существующий из `hyg_v42.csv.gz` → `hyg_v42.csv` → `hyg_v3.csv.gz` → `hyg_v3.csv` в `data/catalogs/` | каталог HYG (закоммиченный `hyg_v42.csv.gz` работает на свежем клоне) |
| `--lines` / `--names` | `STARGLYPH_SERVE_LINES/_NAMES` | репо-копии | геометрия/имена созвездий |
| `--cache-dir` | `STARGLYPH_SERVE_CACHE_DIR` | `prototype/artifacts/cache` | кэш баз (персистентный) |
| `--pool-size` | `STARGLYPH_SERVE_POOL_SIZE` | `2` | прогретых движков = параллельных solve |
| `--max-body-mib` | `STARGLYPH_SERVE_MAX_BODY_MIB` | `32` | лимит загрузки |
| `--solve-timeout-s` | `STARGLYPH_SERVE_SOLVE_TIMEOUT_S` | `120` | таймаут одного solve |
| `--queue-timeout-s` | `STARGLYPH_SERVE_QUEUE_TIMEOUT_S` | `30` | ожидание свободного движка |
| `--prewarm-dense` | `STARGLYPH_SERVE_PREWARM_DENSE` | `22,40,65` | центры dense-полос для фонового прогрева диск-кэша (пусто — выключить) |

Пути по умолчанию считаются от репозитория (как у CLI) — вне репозитория
задавайте `--catalog/--lines/--names/--cache-dir` явно. Контейнеризация и
ресурс-бюджет — C3.

## Модель конкурентности (почему пул, а не мьютекс)

`solve_frame_with_engine` принимает `&mut Engine`: solve может лениво достроить
dense-полосу и дописать её в движок. Поэтому:

- **Пул прогретых `Engine`** (`--pool-size`) за семафором; запрос забирает движок
  в эксклюзив на время solve (в `spawn_blocking`, солвер внутри параллелится
  rayon'ом). Один общий `Mutex<Engine>` сериализовал бы все solve; свежий
  `Engine` на запрос (анти-паттерн из desktop `commands.rs`) — перечитывал бы
  сотни МБ баз каждый раз.
- **Single-flight генерации баз** — в самом `Engine::ensure_kind`
  (`starglyph-core`): процессный лок вокруг «сгенерировать + сохранить»,
  повторная проверка диск-кэша под локом и атомарная публикация файла через
  `tmp + rename`. N движков, одновременно промахнувшихся мимо одной полосы,
  строят её один раз; читатели никогда не видят недописанный файл.
- **Возврат движка гарантирован**: solve-задача отвязана от HTTP-запроса
  (обрыв клиента/таймаут не теряют движок), паника solve заменяет движок свежим
  (базы дозагрузятся из кэша лениво).

Память: каждый движок пула держит свои копии загруженных баз в RAM
(bootstrap ~24 МБ; dense-полосы 19–190 МБ). `--pool-size` подбирайте от RAM и
ядер: solve сам параллелится, поэтому больше ядер/2 движков редко имеет смысл.

## Замер (2026-07-06, тёплый кэш, дефолтный пул 2)

`make serve-smoke` на `data/input/CD_2011-09-19_0000.bmp` (22°, blind):
одиночный solve `total_ms` ≈ 183 мс; overlay-PNG ≈ 1.3 МБ; 6 параллельных
запросов — все `200`, медиана `total_ms` ≈ 190 мс, максимум 210 мс,
mtime/размер файлов кэша не изменились (пересборок нет). Холодный старт с
пустым кэшем: `readyz` красный на время генерации bootstrap (~1 мин),
первый solve после готовности — 198 мс.
